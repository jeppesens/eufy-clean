## bug_007 — `LocalTuyaClient.send_command` silently swallows tinytuya error-dict responses

**Severity:** normal
**File:** `custom_components/robovac_mqtt/api/local_tuya.py:140-152`

`LocalTuyaClient.send_command` only treats raised exceptions as failures, but `tinytuya.set_multiple_values` typically returns an error dict (e.g. `{"Err":"914","Error":"Check device key or version"}`) for transport / auth / sequence problems instead of raising. Writes that the dock never accepted appear successful — the user presses Start, no `HomeAssistantError` reaches HA, and the vacuum never moves. Fix: capture the return value and raise `LocalTuyaError` when it is a dict containing `'Err'` or `'Error'` (mirroring the receive-side check the same file already does at lines 164-167).

### Reasoning

`LocalTuyaClient.send_command` (lines 140-151) does:

```python
try:
    await self._loop.run_in_executor(
        None, self._dev.set_multiple_values, dps
    )
except Exception as e:
    raise LocalTuyaError(...) from e
```

The return value is awaited and discarded. The author's own comment on lines 141-143 acknowledges this trade-off: *"Returns a dict on success / error info dict on failure; we only care about exceptions."* That is precisely the bug — `tinytuya` does not raise on most transport-level failures; it returns an error dict.

In `tinytuya` 1.18.0 (the version pinned in `manifest.json`):

- `Device.set_multiple_values` (`Device.py:99-153`) calls `_send_receive` and returns its result unchanged.
- `XenonDevice._send_receive` returns `error_json(...)` (a dict) without raising on:
  - **Line 580**: `return error_json(sock_result if sock_result else ERR_OFFLINE)` — socket cannot be opened (device offline / wrong IP).
  - **Line 640**: `return error_json(ERR_KEY_OR_VER)` — socket retries exhausted on timeout (wrong local key, wrong protocol version, sequence-number mismatch).
  - **Line 653**: `return error_json(ERR_PAYLOAD)` — decode fails after retries (wrong key produces unparseable replies).
- `error_json` builds dicts like `{"Error": "Check device key or version", "Err": "914", "Payload": ...}`.

This PR explicitly sets `self._dev.set_socketRetryLimit(1)` at `local_tuya.py:106` with a 0 s retry delay — meaning a single missed ACK is enough to push `_send_receive` straight into the `error_json` branch.

The same file already handles this convention correctly on the receive side (`_listen_loop` lines 164-167). The send path simply doesn't apply the same check.

### Fix

```python
try:
    result = await self._loop.run_in_executor(
        None, self._dev.set_multiple_values, dps
    )
except Exception as e:
    raise LocalTuyaError(
        f"Failed to send local Tuya command to {self.device_id}: {e}"
    ) from e
if isinstance(result, dict) and ("Err" in result or "Error" in result):
    raise LocalTuyaError(
        f"Local Tuya {self.device_id} rejected command: {result}"
    )
```

## bug_001 — `LocalTuyaClient`: thread-unsafe concurrent access to `tinytuya.Device` socket

**Severity:** normal
**File:** `custom_components/robovac_mqtt/api/local_tuya.py:140-152`

The background `_listen_loop` runs `self._dev.receive()` in one HA executor thread while `send_command` runs `self._dev.set_multiple_values(dps)` in another — both touching the same TCP socket inside a single `tinytuya.Device` instance, which has no internal locking. Because the listener is blocked in `receive()` ~100% of the time (5 s timeout, immediately re-entered), any user button press collides with it: the listener frequently consumes the ACK packet that `set_multiple_values` is awaiting, causing the send to time out and surface as `LocalTuyaError` ("Failed to send local Tuya command") even though the device received the command, plus possible frame corruption and reconnect-time `self._dev` reassignment races. Fix by wrapping all `_dev` access (send, receive, reconnect) in a single `asyncio.Lock`, or by using a dedicated single-threaded executor so operations queue naturally.

### Reasoning

The new local-Tuya transport runs two operations against the **same** `tinytuya.Device` instance from **different** HA executor threads, without any synchronization. Both eventually call into `tinytuya._send_receive`, which directly does `self.socket.sendall(...)` followed by `self._receive()` with no locking — `tinytuya` 1.18.0 has zero `threading.Lock`/`RLock` anywhere in its core package; it is documented as one-thread-per-`Device`.

**The two contending paths:**

- `_listen_loop` (line 158-159) → `_receive_with_timeout` → `self._dev.receive()` running in one executor thread.
- `send_command` (line 145-147) → `self._dev.set_multiple_values(dps)` running in another.

Both are dispatched via `loop.run_in_executor(None, ...)` which uses HA's **default** `ThreadPoolExecutor` (multiple workers — CPython default `min(32, cpu_count+4)`), so they genuinely run concurrently. `socket.sendall`/`socket.recv` release the GIL.

### Concrete failure modes

1. **ACK consumption (most common).** The listener thread, already blocked in `_receive()`, frequently wins the race and consumes the ACK first. With `socketRetryLimit=1`, the writer's retry quickly gives up and surfaces as `LocalTuyaError("Failed to send local Tuya command to %s")` — **even though the device received and executed the command**.
2. **Reconnect-time `self._dev` reassignment.** The listener's reconnect path at lines 174 and 189 reassigns `self._dev` from a different thread via `_open_device`. A concurrent `send_command` may capture the old (closed) `_dev`, or observe a half-constructed new one.
3. **Framing corruption.** Concurrent `socket.recv()`/`sendall()` from two threads on a single TCP socket can interleave reads, splitting a single Tuya frame across two readers.
4. **`set_socketTimeout` clobbering.** Two threads can leave the socket with the wrong timeout for whichever caller acquires it next.

### Fix

Serialize all `_dev` access. Options:

- `asyncio.Lock` on the `LocalTuyaClient` wrapping receive, send, and reconnect (caveat: holds the lock for up to 5 s during a receive).
- A dedicated **single-threaded** `ThreadPoolExecutor(max_workers=1)` per `LocalTuyaClient` (operations queue naturally; cleanest).
- An asyncio sentinel that lets `send_command` short-circuit the in-flight receive.

## bug_003 — Cloud poll: empty `dps {}` dict counted as failure, triggers spurious unavailability

**Severity:** normal
**File:** `custom_components/robovac_mqtt/coordinator.py:360-381`

Cloud polling treats an empty DPS dict (`{}`) as a failure: `if dps:` in `coordinator._async_update_data` is falsy for both `None` and `{}`, but `TuyaCloudClient.get_device` returns `device.get("dps", {})` — i.e. `{}` for a found-but-stateless device. After 5 such successful-but-empty polls (~2.5 min), the entity is marked unavailable and exponential backoff kicks in even though the cloud is healthy. Fix is one line: `if dps is not None:`.

### Reasoning

```python
dps = await self.eufy_login.getCloudDevice(self.device_id)
if dps:                          # <-- truthy check
    new_state, _ = self._parse_dps(dps)
    self._on_cloud_success()
    return new_state
...
self._on_cloud_failure()
```

`TuyaCloudClient.get_device` in `api/tuya_cloud.py:237-246`:

```python
for device in devices:
    if device.get("devId") == device_id:
        dps = device.get("dps", {})   # <-- defaults to {}, not None
        return dps
```

Trigger is realistic: a freshly registered device, a vacuum that hasn't pushed state since onboarding, a brief Tuya-side cache miss for an idle device. Impact: poll interval doubled on first "failure"; after 5 consecutive empty polls (~2.5 min), `UpdateFailed` raised and entity shown unavailable.

### Fix

```python
dps = await self.eufy_login.getCloudDevice(self.device_id)
if dps is not None:
    new_state, _ = self._parse_dps(dps)
    self._on_cloud_success()
    return new_state
```

`_parse_dps({})` is already a no-op.

## bug_010 — `tuya_login`: EU→US fallback only triggers on `TuyaCloudError`, not network errors

**Severity:** normal
**File:** `custom_components/robovac_mqtt/api/cloud.py:83-91`

The EU-region Tuya Cloud login attempt in `tuya_login()` only catches `TuyaCloudError`, but the underlying HTTP request can raise `aiohttp.ClientError` (network/DNS), `asyncio.TimeoutError` (15 s timeout), `KeyError` (unexpected response shape), or `ValueError` (`int(exponent)` on bad data). Any of these propagate past the EU `except` block and skip the documented US fallback entirely — a single transient EU outage takes Tuya Cloud out for the whole HA session and all legacy cloud devices go unavailable until reload.

### Reasoning

```python
try:
    client = TuyaCloudClient("EU", websession=self._websession)
    await client.login(self._eufy_user_id)
    self.tuya_client = client
    return
except TuyaCloudError as e:
    _LOGGER.debug("Tuya Cloud EU login failed: %s", e)

# Fall back to US
try:
    client = TuyaCloudClient("US", ...)
    ...
```

Inside `TuyaCloudClient.login()` → `TuyaCloudClient.request()` (api/tuya_cloud.py:197-202):

```python
async with session.get(self.endpoint, params=params, timeout=_REQUEST_TIMEOUT) as resp:
    body = await resp.json(content_type=None)
```

Can raise `aiohttp.ClientError` and subclasses, `asyncio.TimeoutError`, `aiohttp.ContentTypeError`, `json.JSONDecodeError`. Login post-processing can raise `KeyError` / `ValueError`. None inherit from `TuyaCloudError`.

`init()` only catches `Exception` to log a warning — it cannot tell the EU branch failed for a network reason and that the US branch was never attempted.

### Fix

```python
except (TuyaCloudError, aiohttp.ClientError, asyncio.TimeoutError, OSError, KeyError, ValueError) as e:
    _LOGGER.debug("Tuya Cloud EU login failed: %s", e)
```

Or simply `except Exception` for the EU attempt.

## merged_bug_002 — OptionsFlow silently wipes local-Tuya overrides for devices not currently running

**Severity:** normal
**File:** `custom_components/robovac_mqtt/config_flow.py:205-221`

`OptionsFlowHandler.async_step_init` builds `new_overrides` only from devices in `eligible` (live coordinators), then writes that as the *entire* `CONF_LOCAL_DEVICES` value — silently destroying any saved override whose coordinator isn't currently running (transient cloud outage, failed `coordinator.initialize()`, model dropped by `findModel`, or `ConfigEntryNotReady` with empty `coordinators`). In the worst case (login failed → `coordinators == []`), opening Configure once and clicking Submit wipes **every** saved LAN host for the entry, with no UX hint and no recovery once the cloud comes back. Fix is one line: seed `new_overrides = dict(existing)` (or merge non-eligible entries) before iterating `eligible`.

### Reasoning

```python
if user_input is not None:
    new_overrides: dict[str, dict] = {}
    for dev_id, _name, _model in eligible:
        ...
        if host:
            new_overrides[dev_id] = {...}
    return self.async_create_entry(
        title="",
        data={**self._entry_ref.options, CONF_LOCAL_DEVICES: new_overrides},
    )
```

`new_overrides` starts empty and only grows with entries from `eligible`, then *replaces* the entire `CONF_LOCAL_DEVICES` dict in options. There is no merge with the existing dict.

The `eligible` list only includes devices that have a live coordinator. Realistic ways a coordinator goes missing while options survive:

1. **Tuya cloud transient failure** — `getCloudDevices()` (`api/cloud.py:144`) catches `TuyaCloudError`, logs a warning and returns early.
2. **Per-coordinator init failure** — `__init__.py:118-121` catches and logs.
3. **Auth/network failure during setup** — entry in SETUP_ERROR/SETUP_RETRY, `coordinators` is `[]`.
4. **Model not in `EUFY_CLEAN_DEVICES`** — `findModel()` filters it out.

Worst-case proof: user has overrides for three docks. HA restart with Tuya cloud briefly down → `coordinators = []`. User opens Configure → empty form → Submit → `user_input = {}` → loop is no-op → empty `new_overrides` written, destroying all three saved overrides silently.

### Fix

```python
if user_input is not None:
    existing = self._entry_ref.options.get(CONF_LOCAL_DEVICES, {})
    new_overrides: dict[str, dict] = {
        k: v for k, v in existing.items()
        if k not in {dev_id for dev_id, _, _ in eligible}
    }
    for dev_id, _name, _model in eligible:
        ...
```

Or include orphaned-but-configured devices in `eligible` with a "currently unavailable" label.

## bug_013 — `_initialize_local`: `LocalTuyaError` from constructor escapes fallback path

**Severity:** nit
**File:** `custom_components/robovac_mqtt/coordinator.py:166-181`

`_initialize_local` constructs `LocalTuyaClient` (line 161-166) and calls `set_on_message` (line 167) outside the try block — only `client.connect()` is wrapped. `LocalTuyaClient.__init__` raises `LocalTuyaError` when `tinytuya is None` (api/local_tuya.py:57-60), so a missing/unimportable tinytuya bypasses the documented fallback-to-cloud branch and propagates up to `async_setup_entry`'s broad except, silently dropping the coordinator. Fix is a one-line move: shift the `LocalTuyaClient(...)` construction inside the try at line 168 so the same fallback runs.

### Reasoning

`_initialize_local` (coordinator.py:137-181) is documented to fall back to cloud polling "if the local socket can't be opened". The fallback is `try / except (LocalTuyaError, OSError, TimeoutError)` — but the `try` only starts at line 168, around `await self.client.connect()`. The `LocalTuyaClient(...)` constructor at line 161 and the `self.client.set_on_message(...)` call at line 167 sit *outside* the `try`.

`LocalTuyaClient.__init__` raises `LocalTuyaError` when the module-level `tinytuya` import resolved to `None` (api/local_tuya.py:57-60). The import is wrapped in `try / except ImportError` at lines 27-30, so the module loads even when tinytuya is unavailable, but any attempt to construct a client then fails.

Compare with `_initialize_mqtt` at coordinator.py:183 — its construction of `EufyCleanClient` *is* inside its try block.

Trigger conditions are narrow but real:

1. HACS install before HA restart (requirements not pip-installed yet).
2. Broken venv / partial install.
3. Permission / filesystem issues preventing pip from completing.

### Fix

```python
try:
    self.client = LocalTuyaClient(
        device_id=self.device_id,
        local_key=self._local_key,
        host=self._local_host,
        version=self._local_version,
    )
    self.client.set_on_message(self._handle_mqtt_message)
    await self.client.connect()
except (LocalTuyaError, OSError, TimeoutError) as e:
    ...
```
